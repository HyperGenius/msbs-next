# テスト用アカウント仕様

## 概要

オンボーディングフローや初期状態からのゲームプレイを繰り返しテストするための、アカウントリセット機能について記述します。

## 目的

- 新規ユーザー体験（オンボーディング）の動作確認を容易にする。
- 開発・テスト環境において、特定の状態をクリアしてクリーンな状態でテストを開始できるようにする。

---

## Backend: データ削除API

### エンドポイント

```
DELETE /api/pilots/me
```

**認証**: Clerk JWT Bearer トークンが必要（`Depends(get_current_user)`）

**処理内容**: 認証済みユーザーの `user_id` に紐づく以下のレコードを、トランザクション内でアトミックに削除する。

| 削除順序 | テーブル | 理由 |
|---|---|---|
| 1 | `battle_entries` | 外部キー制約（`mobile_suits.id` 参照）のため先に削除 |
| 2 | `battle_results` | ユーザーの戦績ログ |
| 3 | `mobile_suits` | 所有している全機体 |
| 4 | `pilots` | パイロット基本情報 |

**レスポンス（成功時）**:

```json
{
  "message": "アカウントを初期状態にリセットしました。"
}
```

**レスポンス（エラー時）**: HTTP 500 とエラー詳細を返す。

### 削除後の挙動

- `GET /api/pilots/me` へ再アクセスすると、パイロットが存在しないため HTTP 404 が返る。
- フロントエンドはこの 404 をオンボーディング未完了として扱い、チュートリアル画面を表示する。
- `POST /api/pilots/register` を呼び出すことで、新しいパイロットとスターター機体が付与される。

---

## Frontend: デバッグ用UI

### 表示条件

`process.env.NODE_ENV === "development"` の場合のみ表示される。

### 配置場所

`frontend/src/components/Header.tsx` の `SignedIn` ブロック内（デスクトップナビゲーション）。

### 動作フロー

1. 「Reset Account (Debug)」ボタンをクリック。
2. ブラウザ標準の `confirm()` ダイアログを表示（誤操作防止）。
3. 確認後、`DELETE /api/pilots/me` を呼び出す。
4. 成功後、`localStorage.removeItem("msbs_onboarding_completed")` を実行してローカルのオンボーディング完了フラグをクリア。
5. `window.location.reload()` でページをリロードし、オンボーディング画面へ誘導。
6. エラー時は `alert()` でエラーメッセージを表示する。

---

## セキュリティに関する注意事項

> [!WARNING]
> この機能は非常に強力であり、呼び出すとユーザーの全データが削除されます。

- **本番環境では絶対に使用しないこと。** `NODE_ENV === "development"` の場合のみUIが表示される。
- 将来的に本番環境へデプロイする際は、環境変数（例: `NEXT_PUBLIC_ENABLE_DEBUG_FEATURES=true`）で制御するか、特定の管理者権限を持つユーザーのみに制限することを推奨する。
- APIエンドポイント自体は認証済みユーザーの自己データのみを削除するため、他ユーザーのデータには影響しない。
